Card definitions live in `dashboardMetrics()` in [`src/lib/devcenter/sample.ts:191-219`](../src/lib/devcenter/sample.ts). The page (`src/app/(app)/admin/dev-command/page.tsx`) is a thin wrapper: in `MOCK_MODE` it calls `dashboardMetrics()` with no overrides (pure sample data); otherwise it calls `dashboardMetrics(live.counts)`, where `live.counts` comes from `getLiveDashboardData()` in [`src/lib/devcenter/repo.ts:278-343`](../src/lib/devcenter/repo.ts). ## Open questions / unconfirmed windows

- **`runs_today` window differs by mode.** `page.tsx` maps `runs_today` to `live.counts.runningAgents`, which is `repo.ts:296-297`'s live count of `dev_agent_runs` with `status = 'running'` — a point-in-time snapshot, not a "today" count. But the sample/mock fallback in `sample.ts:198` filters by `isToday(created_at)`. In practice, production (non-`MOCK_MODE`) traffic always uses the live "running" snapshot, so the label is written for that behavior ("active right now"). If `MOCK_MODE` is ever used to demo this specific card, the label will describe live behavior, not the mock behavior on screen. Flagging for reviewer awareness — not blocking, since neither number/query changed.
- **`active_prs`, `recent_deploys`, and `audit_today` are never wired to live data**, in `MOCK_MODE` or not — `dashboardMetrics()` only accepts overrides for `open_tasks`, `need_approval`, `runs_today`, `failed_runs`, `security_warnings`, `xp_failures`, and `draft_plans` (see the `live` param destructuring in `sample.ts:191-194`, and the keys passed from `page.tsx:37-45`). These three cards always show `SAMPLE_DEPLOYMENTS`/`SAMPLE_AUDIT` fixture data even when real tasks exist. The *window* each card claims (currently open / all time / today) is confirmed correct against the sample-data filter that actually produces the number today — but the number itself is not live-backed. This looks like a pre-existing gap unrelated to labeling; out of scope for this copy-only change, called out here for a human decision on whether to file a follow-up.
